# Liskov Substitution Principle (LSP)

Subtypes must be **substitutable** for their base types: code that works with the parent should keep working when you pass a subclass, **without surprises** or broken invariants.

## Real-life analogy

Imagine a pet hotel with a policy: every guest must be **fed**, **walked**, and **groomed**. Dogs, cats, and rabbits mostly fit.

**Problem:** Someone brings a **snake**. Walking and grooming do not apply; feeding is not “pet food” in the same sense. The hotel’s workflow assumed “pet ≈ dog-like,” so substituting a snake **breaks** those assumptions. That is an LSP violation: the subtype does not honor the contract implied by the base concept in this design.

**Contrast:** A **hamster** might use a wheel instead of a walk, but you can still fulfill “care” in a predictable way without invalidating the overall process.

## Classic violation: `Square` extends `Rectangle`

A common teaching example: if `Square` **is-a** `Rectangle` in code, then any function written for `Rectangle` must behave correctly when given a `Square`. Often it does not.

**What goes wrong:** `printArea` sets width to 5 and height to 10, expecting area **50**. For a plain `Rectangle`, that holds. For a `Square`, `setHeight(10)` also forces width to 10 (to stay square), so area becomes **100**. The **postcondition** of “independent width and height” that `Rectangle` suggests is violated by `Square`.

```cpp
#include <iostream>
using namespace std;

class Rectangle {
protected:
    int width, height;

public:
    virtual void setWidth(int w) { width = w; }
    virtual void setHeight(int h) { height = h; }
    int getArea() { return width * height; }
};

class Square : public Rectangle {
public:
    void setWidth(int w) override {
        width = w;
        height = w;
    }

    void setHeight(int h) override {
        height = h;
        width = h;
    }
};

void printArea(Rectangle* r) {
    r->setWidth(5);
    r->setHeight(10);
    cout << r->getArea() << endl;  // For Rectangle: 50. For Square: 100 (unexpected).
}

int main() {
    Rectangle* r = new Square();
    printArea(r);
    delete r;
    return 0;
}
```

**Why this violates LSP:** Subclasses must not **weaken** preconditions or **strengthen** requirements in ways that break callers. Here, substituting `Square` changes the meaning of “set width then set height” compared to `Rectangle`.

**Better directions (conceptually):** do not model `Square` as a subtype of mutable `Rectangle` if the API implies independent sides. Alternatives include a common `Shape` interface, **composition**, or immutable shapes with factories—so contracts stay clear and substitutable.

## Valid substitution: notifications

Here, `EmailNotification` and `TextNotification` override `sendNotification()` in a way that still satisfies “you can send a notification through the base pointer.” Callers depend on the **base contract**; behavior may differ, but the operation remains valid.

```cpp
#include <iostream>
using namespace std;

class Notification {
public:
    virtual void sendNotification() {
        cout << "Notification sent" << endl;
    }

    virtual ~Notification() = default;
};

class EmailNotification : public Notification {
public:
    void sendNotification() override {
        cout << "Email notification sent" << endl;
    }
};

class TextNotification : public Notification {
public:
    void sendNotification() override {
        cout << "Text notification sent" << endl;
    }
};

int main() {
    Notification* notification = new EmailNotification();
    notification->sendNotification();
    delete notification;
    return 0;
}
```

## Interview-style summary

| Idea | One-liner |
|------|-----------|
| **LSP** | Subtypes must honor the **contract** (behavioral expectations) of the base type. |
| **Rectangle / Square** | Inheritance that changes **invariants** of the parent API often breaks LSP. |
| **Fix** | Prefer design where the abstraction matches the real subtype rules (interfaces, composition, clear contracts). |
| **Related** | Often discussed with **Design by Contract** (preconditions, postconditions, invariants). |
