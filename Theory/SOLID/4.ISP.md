# Interface Segregation Principle (ISP)

A client should **not** be forced to depend on methods it does **not** use. Prefer **small, focused** interfaces over one large “kitchen sink” interface.

## Real-life analogy

At a restaurant, if you only want vegetarian dishes, you should not be forced to page through or “depend on” the entire non-vegetarian section. Menus are often split into sections so each diner uses what matters to them.

## Real-world example: ride-hailing app

### Bad design (violates ISP)

One fat interface mixes **rider** and **driver** responsibilities. A `Rider` must still declare methods that only drivers use—often as no-ops, `UnsupportedOperationException`, or fake implementations. That is a smell: the type is **forced** to know about operations it should not expose.

```java
/** Too many unrelated responsibilities in one contract. */
public interface UberUser {
    void bookRide();

    void acceptRide();

    void trackEarnings();

    void ratePassenger();

    void rateDriver();
}

/** Rider is forced to implement driver-only methods. */
public class Rider implements UberUser {

    @Override
    public void bookRide() {
        /* real logic */
    }

    @Override
    public void rateDriver() {
        /* real logic */
    }

    @Override
    public void acceptRide() {
        throw new UnsupportedOperationException("Riders don't accept rides");
    }

    @Override
    public void trackEarnings() {
        throw new UnsupportedOperationException("Not a driver");
    }

    @Override
    public void ratePassenger() {
        throw new UnsupportedOperationException("Riders don't rate passengers");
    }
}
```

### Good design (follows ISP)

Split contracts by **role**. Each client depends only on the methods it actually uses.

```java
/** Operations a passenger / rider needs. */
public interface RiderActions {
    void bookRide();

    void rateDriver();
}

/** Operations a driver needs. */
public interface DriverActions {
    void acceptRide();

    void trackEarnings();

    void ratePassenger();
}

public class Rider implements RiderActions {

    @Override
    public void bookRide() {
        /* … */
    }

    @Override
    public void rateDriver() {
        /* … */
    }
}

public class Driver implements DriverActions {

    @Override
    public void acceptRide() {
        /* … */
    }

    @Override
    public void trackEarnings() {
        /* … */
    }

    @Override
    public void ratePassenger() {
        /* … */
    }
}
```

If a single user could be both rider and driver in your domain, that person can implement **both** small interfaces—without forcing every rider to implement driver methods.

## Benefits

1. **Modularity and flexibility** — Clients depend on narrow contracts; changes to driver-only APIs do not ripple to rider-only code.
2. **Easier testing** — Mock or stub small interfaces in unit tests without unused methods.
3. **Prevents bogus implementations** — Fewer “throw unsupported” or empty methods that hide design mistakes.

## When to apply ISP

1. When an interface **does too many unrelated things** (multiple roles or concerns in one type).
2. When implementers **throw unsupported**, return dummy values, or leave methods empty just to satisfy the interface.
3. When tests or mocks repeatedly stub methods that the class under test never uses.

## Interview-style summary

| Topic | Takeaway |
|-------|----------|
| **ISP** | Many small interfaces > one fat interface; clients see only what they need. |
| **vs SRP** | SRP is about **one responsibility per class**; ISP is about **splitting surface area** of types clients depend on. |
| **Trade-off** | More interfaces to name and maintain; use when roles are genuinely different. |
