# Open/Closed Principle

A class should be **open for extension** but **closed for modification**.

You can add new behavior without changing existing, stable code.

## Real-life Analogy

Power adapters: you travel from India to the UK. Your Indian charger does not fit UK sockets. You use a **travel adapter** so the same charger works abroad. You extended usability; you did **not** modify the charger itself.

In code, OCP favors adding functionality through **extension** (new types, new implementations) instead of editing working logic repeatedly.

## Real-World Example: Region-Based Tax

An invoicing system must apply tax by region (illustrative rates):

| Region | Tax   |
|--------|-------|
| India  | GST 18% |
| US     | Sales tax 8% |
| UK     | VAT 12% |

New regions may be added over time.

### Bad design (violates OCP)

One method grows every time a region is added: **if / else if** chains and magic numbers.

```java
public class InvoiceProcessor {

    public double calculateTotal(String region, double amount) {
        if ("India".equals(region)) {
            return amount + amount * 0.18;
        } else if ("US".equals(region)) {
            return amount + amount * 0.08;
        } else if ("UK".equals(region)) {
            return amount + amount * 0.12;
        } else {
            return amount; // no tax for unknown region
        }
    }
}
```

**Why this is problematic**

- Adding a region (e.g. Germany) means **editing** `calculateTotal` and risks breaking existing branches.
- Hard to unit-test each rule in isolation.
- Violates OCP: behavior changes by **modifying** the same class.

### Good design (follows OCP)

Extract each tax rule behind a **strategy** (`TaxCalculator`). `Invoice` depends on the **abstraction**, not on region strings.

**Extension:** add a new class (e.g. `GermanyTaxCalculator`) implementing `TaxCalculator`.  
**Closed for modification:** `Invoice` and the interface stay unchanged.

```java
/** Strategy: one implementation per region / rule set. */
public interface TaxCalculator {
    double calculateTax(double amount);
}

public final class IndiaTaxCalculator implements TaxCalculator {
    @Override
    public double calculateTax(double amount) {
        return amount * 0.18; // GST (illustrative)
    }
}

public final class USTaxCalculator implements TaxCalculator {
    @Override
    public double calculateTax(double amount) {
        return amount * 0.08; // sales tax (illustrative)
    }
}

public final class UKTaxCalculator implements TaxCalculator {
    @Override
    public double calculateTax(double amount) {
        return amount * 0.12; // VAT (illustrative)
    }
}

public final class Invoice {
    private final double amount;
    private final TaxCalculator taxCalculator;

    public Invoice(double amount, TaxCalculator taxCalculator) {
        this.amount = amount;
        this.taxCalculator = taxCalculator;
    }

    public double getTotalAmount() {
        return amount + taxCalculator.calculateTax(amount);
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        double amount = 1000.0;

        Invoice indiaInvoice = new Invoice(amount, new IndiaTaxCalculator());
        System.out.println("Total (India): " + indiaInvoice.getTotalAmount());

        Invoice usInvoice = new Invoice(amount, new USTaxCalculator());
        System.out.println("Total (US): " + usInvoice.getTotalAmount());

        Invoice ukInvoice = new Invoice(amount, new UKTaxCalculator());
        System.out.println("Total (UK): " + ukInvoice.getTotalAmount());
    }
}
```

### Interview-style summary

| Topic | Bad design | Good design |
|-------|------------|-------------|
| **OCP** | Change one big method for every new region | Add a new `TaxCalculator` implementation |
| **Testing** | One method, many branches | Test each calculator separately |
| **Coupling** | `InvoiceProcessor` knows all regions | `Invoice` knows only `TaxCalculator` |

**Common techniques for OCP:** Strategy, Template Method, plugin-style interfaces, dependency injection.

**Trade-off:** more classes and indirection; worth it when the domain changes often or many variants exist.
