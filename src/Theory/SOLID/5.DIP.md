# Dependency Inversion Principle (DIP)

High-level modules should **not** depend on low-level modules.  
Both should depend on **abstractions**.

In simple words: business logic should depend on an interface/contract, not on a concrete implementation.

## Real-life analogy

Think of a phone charger and wall socket:
- Your phone depends on a **charging standard** (USB-C), not on one specific charger brand.
- Any charger that follows USB-C can be used without changing the phone.

That "standard" is the abstraction. DIP asks us to design code the same way.

## Example: recommendation engine

### Bad design (violates DIP)

The high-level class `RecommendationEngine` directly depends on one concrete class (`RecentRecommendation`).  
If we want to switch to personalized or hybrid recommendation, we must modify `RecommendationEngine`.

```java
class RecentRecommendation {
    public void recommend() {
        System.out.println("Getting recent recommendation");
    }
}

class PersonalizedRecommendation {
    public void recommend() {
        System.out.println("Getting personalized recommendation");
    }
}

class HybridRecommendation {
    public void recommend() {
        System.out.println("Getting hybrid recommendation");
    }
}

/** High-level module tightly coupled to one low-level class. */
class RecommendationEngine {
    private final RecentRecommendation recentRecommendation;

    public RecommendationEngine() {
        this.recentRecommendation = new RecentRecommendation();
    }

    public void recommend() {
        recentRecommendation.recommend();
    }
}
```

### Good design (follows DIP)

Introduce an abstraction (`RecommendationStrategy`) and make all concrete strategies implement it.  
Now `RecommendationEngine` depends only on the interface, and strategy can be swapped via constructor injection.

```java
interface RecommendationStrategy {
    void recommend();
}

class RecentRecommendation implements RecommendationStrategy {
    @Override
    public void recommend() {
        System.out.println("Getting recent recommendation");
    }
}

class PersonalizedRecommendation implements RecommendationStrategy {
    @Override
    public void recommend() {
        System.out.println("Getting personalized recommendation");
    }
}

class HybridRecommendation implements RecommendationStrategy {
    @Override
    public void recommend() {
        System.out.println("Getting hybrid recommendation");
    }
}

/** High-level module now depends on abstraction. */
class RecommendationEngine {
    private final RecommendationStrategy recommendationStrategy;

    public RecommendationEngine(RecommendationStrategy recommendationStrategy) {
        this.recommendationStrategy = recommendationStrategy;
    }

    public void recommend() {
        recommendationStrategy.recommend();
    }
}

class Main {
    public static void main(String[] args) {
        RecommendationEngine engine =
                new RecommendationEngine(new PersonalizedRecommendation());
        engine.recommend();
    }
}
```

## Benefits of DIP

1. **Loose coupling** - High-level policy code does not break when low-level details change.
2. **Easy extension** - Add new strategies without modifying engine logic.
3. **Better testability** - Pass mocks/stubs of `RecommendationStrategy` in unit tests.

## Interview-style summary

| Topic | Takeaway |
|-------|----------|
| **DIP** | Depend on interfaces/abstractions, not concrete classes. |
| **High-level module** | Contains business rules/policy. |
| **Low-level module** | Contains implementation details (DB/API/file/algorithm specifics). |
| **Common technique** | Constructor injection with interfaces. |
